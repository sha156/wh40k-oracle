# LLM PDF 重构设计（提升兵牌检索准确率）

日期：2026-07-02
状态：已确认（试点先行）

## 背景与问题

当前 RAG 流程（`ingest.py`）用 PyMuPDF 提取 PDF 文本后直接语义分块。战锤兵牌页
（datasheet）的属性表和武器表被"拍扁"成一维文字流，表头（射程/A/BS/S/AP/D、
M/T/SV/W/LD/OC）与数值分离，SemanticChunker 还可能把一个单位切成两半。
检索时 LLM 拿到的是没有上下文的数字，导致回答单位属性/武器数据不准确；
规则文字页也存在结构丢失问题。

已验证事实：

- `data/` 下 49 个 PDF 中 47 个有完整文字层，仅《战锤40K总规则10版老湿腐版1.11》
  和《死亡守卫10版中文老湿腐版1.1》部分页面无文本。
- 文字层数值精确、顺序正确，只是丢失表格结构 → 文本 LLM 重排即可，无需 OCR/视觉识别。
- 现有 LLM 接口：deepseek-chat / glm-4-flash（OpenAI 兼容）。
- `data/` 存在多组新旧版本共存（艾达灵族 1.13/1.2、钛帝国 0115/1112、吞世者×3、
  混沌恶魔×3 等），新旧冲突内容同时被检索会加剧答案错乱。

## 方案选型

已对比：MinerU（73k★，中文文档解析强但安装重、CPU 慢、不懂领域分块）、
marker（37k★）、Docling（62k★）、markpdfdown（纯 VLM）。
结论：文字层完好的前提下，**文本 LLM + 战锤领域 schema 重构**比通用版面模型
更贴合，且一次性成本极低（全量约 ¥3-5）。通用工具无法解决"一个单位 = 一个
chunk"的领域分块问题，仍需自写后处理，故不采用。

**采用方案：deepseek-chat 离线重构 + 按单位/章节结构化分块 + 旧版归档。**
无文本页后续用 glm-4v-flash（免费 VLM）渲染图片兜底（非试点范围）。

## 架构

```
data/*.pdf
   │  llm_refine.py（新增，离线一次性）
   │  PyMuPDF 逐页提取文本 → deepseek-chat 按领域 schema 重排
   ▼
data_refined/<书名>/page_NNN.md      ← 按页内容哈希缓存，可断点续跑
   │  ingest.py（改造）
   │  Markdown 标题分块（## 条目名 为界），替代 SemanticChunker
   ▼
local_vector_store/（FAISS + 元数据 book/unit/page）
```

`app.py` 检索链路（FAISS + BM25 + RRF + FlashRank）不变。

## 组件设计

### 1. llm_refine.py（新增）

- 遍历指定 PDF（支持 `--book` 单本试点、`--all` 全量），逐页提取文本。
- 每页文本附带前一页尾部约 500 字上下文，处理跨页单位的拼接；
  LLM 输出中以标记（如 `<!--CONT-->`）声明"本页开头延续上一条目"，
  合并阶段拼回前一条目。
- Prompt 固定战锤领域 schema，要求输出结构化 Markdown：
  - 兵牌 → `## <单位名>` + 属性表（M/T/SV/W/LD/OC）+ 远程/近战武器表
    （名称/射程/A/BS 或 WS/S/AP/D + 武器技能）+ 单位技能 + 单位构成/分数 + 关键词行
  - 战略技能 → `## <技能名>`（CP 消耗/时机/对象/效果 表格）
  - 强化升级/分队规则 → 对应标题 + 表格
  - 普通规则文字 → 干净 Markdown，保留标题层级
  - 明确禁止：改写数值、增删内容、翻译名词
- 缓存：`data_refined/<书名>/page_NNN.md`，旁存 `page_NNN.meta.json`
  （页文本 SHA-256、prompt 版本号）。页内容和 prompt 版本均未变则跳过。
- 并发调用（asyncio，限速），失败重试 3 次，单页失败落盘原始文本并记录告警，
  不中断整本处理。
- API key 从环境变量 `DEEPSEEK_API_KEY` 读取，缺失时启动即报错。
- 无文本页（<20 字符）：试点阶段跳过并记录清单；二期接 glm-4v-flash 渲染兜底。

### 2. ingest.py 改造

- 新增数据源模式：优先读 `data_refined/<书名>/*.md`（存在则用），
  否则回退现有 PDF 流程（保证未重构的书仍可用）。
- 重构书目的分块：按 `##` 标题切分，一个条目（单位/战略技能/升级）= 一个 chunk；
  超长条目（>约 2000 字符）按 `###` 或段落二次切分，但属性表+武器表永不与
  单位名分离。
- chunk 元数据：`book`、`page`（条目起始页）、`unit`（`##` 标题文本）、`source`。
- 增量记录 `processed_files.json` 兼容现状，重构版内容以
  `data_refined` 目录 mtime/hash 参与增量判断。

### 3. 数据整理（去重归档）

- 新建 `data/archive/`，识别出的旧版本移入（不删除）。执行前列出重复组清单，
  经用户确认后移动。
- `ingest.py` 只扫描 `data/*.pdf`（不递归 archive）。已入库的旧版内容通过
  `--rebuild` 清除。

## 错误处理

- LLM 输出非 Markdown/明显截断：重试；三次失败保留原始页文本入库（降级不丢内容）。
- 数值防篡改抽验：脚本随机抽样 N 页，校验"原始页文本中的数字 token 多重集合
  ⊇ 生成 Markdown 中的数字 token"，不符则标记人工复查。
- API 限流/网络失败：指数退避重试；断点续跑靠页级缓存天然支持。

## 试点与验证（本期范围）

1. 实现 `llm_refine.py`，仅跑《钛帝国十版CODEX-20251112.pdf》（78 页）。
2. 人工抽查 ≥5 个兵牌：原 PDF 页 vs 生成 Markdown 表格逐项比对。
3. 改造 `ingest.py`，用重构结果重建该书索引。
4. 准备 ~10 个已知答案问题（单位属性、武器数据、战略技能各若干），
   新旧索引各问一遍，对比准确率。
5. 试点通过后再全量跑 49 本 + 去重归档 + VLM 兜底（后续迭代）。

## 非目标（YAGNI）

- 不替换检索链路 / 不引入 MinerU 等重型解析器
- 不做英文书与中文书的跨语言对齐
- 不做兵牌图片/插图提取
