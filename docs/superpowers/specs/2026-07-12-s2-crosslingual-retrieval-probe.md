# S2 中英跨语检索实测报告（11 版迁移收尾）

> 2026-07-12。迁移计划 `2026-07-10-edition-11-migration.md` 里唯一显式 ⏳ 项。
> 问题：11 版规则真源是**英文** Core Rules，用户提问是**中文**。BM25 中文分词对英文页
> 必然失配，检索能否靠 bge-m3 跨语 + 别名扩展 + 规则层保底把 11 版英文规则纳入上下文？

## 方法

- 25 条纯中文规则查询，覆盖 S6 大改（掩体/隐蔽/灵能/特殊保护/曲射/近战先攻/接战范围/
  战略预备队/翱翔…）+ 核心机制（速射/致命一击/暴击/战斗休克/攻击序列/致命伤…）。
- 每条查询对比两路，各取 top-8，按 chunk 的 `edition/layer` 元数据统计：
  - **A 纯向量**：`similarity_search`（无查询扩展、无 BM25、无规则层保底）——量 bge-m3 裸跨语。
  - **B 生产**：`hybrid_retrieve`（查询扩展 + FAISS + BM25 + RRF + 规则层保底）——真实链路。
- harness：`scripts/s2_crosslingual_probe.py`（脱 Streamlit，直接装载 bge-m3 + FAISS + BM25，
  `python scripts/s2_crosslingual_probe.py <输出json>` 可复跑）。
- 权威真源 = `layer=rules` 且 `edition=11`（11 版英文 Core Rules，141 个 chunk）。

## 结果

| 指标（25 查询） | 纯向量 A | 生产 B（修复前） | 生产 B（修复后） |
|---|---|---|---|
| top-8 含 `layer=rules` | 14/25 | 23/25 | **25/25** |
| top-8 含 `edition=11` | 20/25 | 24/25 | **25/25** |
| top-1 是 `edition=11` | 12/25 | 4/25 | 4/25 |
| top-1 是 `layer=rules` | 9/25 | 4/25 | 4/25 |

### 关键发现

1. **bge-m3 裸跨语能力真实有效**：纯向量对 20/25 查询能在 top-8 召回 11 版内容、14/25 命中
   规则层、12/25 的 top-1 直接是 11 版英文 Core Rules。中文问 → 英文规则的跨语召回**成立**，
   不是"BM25 必失配 = 检索崩溃"。这推翻了迁移计划风险清单里"中文问答体验会显著下降"的悲观预设。

2. **规则层保底是关键补强**：生产路径的 `RULES_FLOOR_K=2` 保底检索（单独按 `layer=rules`
   过滤取最优 2 条注入上下文）把规则层覆盖从裸跨语的 14/25 提到 23→25/25，确保版本仲裁的
   最高真源始终在场。

3. **BM25 把 top-1 拉向 10 版中文 codex**：生产 top-1 是 11 版仅 4/25——因为 BM25 中文关键词
   强匹配反复出现查询词的 10 版中文 codex 页，RRF 融合把它们顶到前排；规则层保底注入在**队尾**。
   **但这不构成错误**：11 版规则始终在上下文中，版本仲裁 prompt（54e09a2a）指示 LLM 规则冲突
   以 11 版为准，v3 基准 99.0 零硬错佐证 LLM 能正确取舍。检索把"该看的都摆上桌"，取舍交给生成层。

## 根因与修复（本次改动）

修复前有 **2/25 查询（灵能攻击、命中修正上限）生产路径 11 版规则挂零**。逐条追因：

- 规则层保底走 `similarity_search(query, k=2, filter={"layer":"rules"}, fetch_k=200)`。
  langchain FAISS 的 filter 语义是**先取 fetch_k 个向量近邻、再按元数据过滤**。
- 规则层仅 141 个 chunk（占全库 5571 的 2.5%）。冷门英文术语（psychic/命中修正上限）与中文
  查询语义距离大，对应规则 chunk 排在 200 名候选池**之外** → 过滤后为空 → 保底失效。
- 这正是 app.py 既有注释（书目过滤路径）预警过的 "fetch_k 太小、过滤后饥饿" 陷阱，规则层保底
  当初漏了同样的放大。

**修复**（`app.py`）：新增 `RULES_FLOOR_FETCH_K = 8000`（≥ 全库规模），规则层保底改用它。
一次全库向量扫描（FAISS 扁平索引，毫秒级）保证 141 个规则 chunk 全进候选池、取到精确 top-2。
实测：`fetch_k=200→0 条` 修复为 `800→命中灵能页 p81`、`3000→命中命中骰页 p9`；改后
**生产路径 25/25 全部命中规则层与 11 版内容**，两条零覆盖失败清零。检索相关 34 测试绿。

## 裁决：整本中文 refine 暂不需要

迁移计划的兜底预案是"11 版核心规则跑一版中文 refine 入库"。本次实测证明：

- 跨语检索工作良好（修复后 25/25 规则层覆盖），失败案例是 `fetch_k` 饥饿的工程问题，
  已用近零成本的一行参数修复根治，**非 bge-m3 跨语能力不足**。
- 故**不触发**整本中文 refine（那是一次全量 LLM 跑 + --rebuild + 基准回归的重活）。
  保留为可选增强：若后续要改善 top-1 排名体验（让 11 版规则在前排而非队尾），届时中文 refine
  会同时提升 BM25 与向量的同语匹配——但当前答案质量（99.0）不需要它。
